# 大明 PPT Agent Team 架构

> 朕要做 PPT，六部听旨。

这是一套受明代内阁六部制启发的 **Multi-Agent** 架构，专为高质量 PPT 生产而设计。它将复杂的内容创作流程拆解为六个专业化角色，各司其职、流水作业，实现 PPT 的「御制级」交付。

Agent协作流程：深度研究 → 事实核查 → 内容大纲 → 配图生成 → HTML PPT 输出。

**English version →** **[README\_EN.md](./README_EN.md)**

***

## 架构概览

本架构由 **1 个调度中枢** + **5 个专职 Agent** 组成，各司其职，流水线协作：

|  编号 |  角色 |    定位   | 核心职能                   |
| :-: | :-: | :-----: | ---------------------- |
|  0  |  皇帝 |    用户   | 最高决策者，审批关键产物，拥有一票否决权   |
|  1  |  内阁 |   调度中枢  | 任务分配、进度监督、结果汇总、最终交付    |
|  2  | 锦衣卫 |   情报司   | 全网深度研究，搜罗权威信源          |
|  3  |  东厂 |   督察司   | 核查研究报告，确保事实准确、无 AI 幻觉  |
|  4  | 翰林院 |  内容架构师  | 将研究报告转化为 TED 风格 PPT 大纲 |
|  5  |  工部 |   视觉工坊  | 根据大纲生成高质量配图            |
|  6  | 织造局 | PPT 工程师 | 将大纲与配图编织为 HTML PPT 成品  |

***

## 完整工作流

```
皇帝 下旨（主题 + 要求）
  │
  ▼

  │
  ▼
锦衣卫 全网深度研究（3000~5000字）
  │
  ▼ 交付「研究报告_初版.md」
  │
东厂 事实核查 ──→ 有问题？──→ 退回锦衣卫修改
  │                              │
  │（通过）                       ▼
  │                         锦衣卫 修订后重新提交
  ▼
东厂 产出「研究报告_终版.md」
  │
  ▼
皇帝 批阅研究报告 ──→ 不通过？──→ 退回，重新研究
  │
  │（通过）
  ▼
内阁 派发翰林院
  │
  ▼
翰林院 按 TED 3S 原则制作「PPT大纲.md」
  │
  ▼
内阁 同时派发工部 & 织造局
  │
  ├──────────────────────────┐
  ▼                          ▼
工部 生成配图              织造局 等待配图
  │                          │
  ▼                          │
「配图清单.md」+ 图片 ──→────┘
                             │
                             ▼
                      织造局 组装 HTML PPT
                             │
                             ▼ 交付「{主题名}.html」
                             │
                      内阁 最终检查（页面无 bug、内容无误）
                             │
                             ▼
                      皇帝 终审批阅 ✓ / ✗
```

## Agent 角色详述

### 1. 内阁（调度中枢）

> *「运筹帷幄，不沾笔墨」*

* 接收皇帝旨意，拆解为子任务并派发各 Agent

* 监控各 Agent 执行进度，处理异常（超时、冲突、返工）

* 协调工部与织造局之间的配图交接

* 汇总织造局成品，最终检查后向皇帝呈递

**原则**：只管任务派发、过程监督和结果汇报，不亲自执行具体任务。

***

### 2. 锦衣卫（深度研究）

> *「天下之事，无不在探」*

* 全网深度研究，3000\~5000 字，严控信源质量

* 信源可信度分级（高 / 中 / 低），每条标注 URL

* 禁止使用未经验证的二手转述、营销号文章

**交付物**：`研究报告_初版.md` → 东厂

***

### 3. 东厂（事实核查）

> *「宁可错杀，不可放过」*

* 对核心发现、数据与案例进行多方交叉验证（≥ 2 个独立信源）

* 检测 AI 幻觉：虚构引用、伪造数据、失效 URL

* 核查不通过则退回锦衣卫修改

**交付物**：`研究报告_终版.md` → 皇帝批阅 → 内阁

***

### 4. 翰林院（内容架构师）

> *「以笔为剑，以纲为骨」*

根据终版研究报告，遵循 **TED 3S 原则**制作 PPT 大纲：

| 原则             | 核心要求         | 执行标准              |
| -------------- | ------------ | ----------------- |
| Story（故事）      | PPT 承载演讲的故事线 | 开篇有钩子，中间有情节，结尾有落点 |
| Simplicity（简洁） | 绝不堆满文字       | 每页文字不超过 6 行，大量留白  |
| Structure（结构）  | 帮观众抓逻辑       | 标题承上启下，视觉线索强化逻辑关系 |

**交付物**：`PPT大纲.md` → 工部 & 织造局

***

### 5. 工部（视觉工坊）

> *「一图胜千言，三秒定乾坤」*

* 根据大纲中标注「建议配图：是」的页面生成配图

* 质量标准：一眼抓人、三秒懂意、情绪张力强

* 不生成剪贴画、低质图库、与主题无关的装饰图

**交付物**：配图文件 + `配图清单.md` → 织造局

***

### 6. 织造局（PPT 工程师）

> *「图文合璧，匠心独运」*

* 调用 `guizang-ppt-skill`，将大纲与配图组装为 HTML PPT

* 遵循"少即是多"视觉原则，每页只传递一个核心信息

* 自测后交付内阁

**交付物**：`{主题名}.html` → 内阁

***

## 交付物清单

|  阶段 | 交付物              | 产出方 |      接收方      |
| :-: | ---------------- | :-: | :-----------: |
|  研究 | `研究报告_初版.md`     | 锦衣卫 |       东厂      |
|  核查 | `研究报告_终版.md`     |  东厂 |    皇帝 → 内阁    |
|  大纲 | `PPT大纲.md`       | 翰林院 | 内阁 → 工部 & 织造局 |
|  配图 | 图片文件 + `配图清单.md` |  工部 |      织造局      |
|  成品 | `{主题名}.html`     | 织造局 |    内阁 → 皇帝    |

## 皇帝审批节点

仅 **2 个关键节点** 需要用户亲自批阅，其余环节由内阁全权调度：

1. **研究报告终版** — 确认研究方向和内容准确性
2. **最终 PPT** — 终审成品质量，决定是否通过

## 支持平台

本架构的角色设定（`AGENTS.md`）可直接移植到以下 Multi-Agent 平台：

| 平台                                        | 说明                                                |
| ----------------------------------------- | ------------------------------------------------- |
| [MiniMax Agent](https://agent.minimax.io) | 将各角色的 `AGENTS.md` 内容作为 Agent 系统提示词导入              |
| [Codex](https://codex.openai.com)         | 将提示词配置为 Codex Agent 的 system prompt，配合工具调用完成研究与生成 |
| [WorkBuddy](https://workbuddy.ai)         | 在 WorkBuddy 工作流中为每个环节创建对应 Agent，粘贴对应提示词           |
| [OpenClaw](https://openclaw.ai)           | 在 OpenClaw 多 Agent 编排界面中按角色编号依次配置                 |
| [Hermes](https://hermes.ai)               | 以 Hermes Agent Chain 模式运行，每个节点对应一个角色提示词           |

> **通用原则**：每个目录下的 `AGENTS.md` 即该角色的完整系统提示词，可直接粘贴至任意支持自定义系统提示的 Agent 平台。

## 快速开始

### 第一步：克隆仓库

```bash
git clone https://github.com/your-username/DaMing-PPT-Agent-Team.git
cd DaMing-PPT-Agent-Team
```

### 第二步：配置各 Agent

在你的 Agent 平台中依次创建以下 6 个 Agent，并将对应目录下的 `AGENTS.md` 内容粘贴为系统提示词：

| 创建顺序 | Agent 名称 | 提示词文件             |
| :--: | -------- | ----------------- |
|   1  | 内阁       | `1-内阁/AGENTS.md`  |
|   2  | 锦衣卫      | `2-锦衣卫/AGENTS.md` |
|   3  | 东厂       | `3-东厂/AGENTS.md`  |
|   4  | 翰林院      | `4-翰林院/AGENTS.md` |
|   5  | 工部       | `5-工部/AGENTS.md`  |
|   6  | 织造局      | `6-织造局/AGENTS.md` |

### 第三步：配置工具权限

根据各 Agent 职能，建议开启以下工具权限：

| Agent | 建议工具                                           |
| ----- | ---------------------------------------------- |
| 锦衣卫   | 网络搜索（Exa / Tavily / Brave Search）              |
| 东厂    | 网络搜索、URL 访问验证                                  |
| 工部    | 图像生成（Image 2/Seedream 5/Banana 2 / Midjourney） |
| 织造局   | 文件读写、HTML 渲染                                   |

### 第四步：启动流程

向**内阁** Agent 发送如下指令即可启动完整流程：

```
主题：[你的PPT主题]
要求：[补充要求，可选]
```

内阁将自动调度其余五个 Agent 完成全流程，并在 2 个关键节点请你审批。

## 关于本项目

本项目的整体架构由冷逸原创开发，依赖 `guizang-ppt-skill` 生成 HTML PPT。

架构设计理念：

* **职责单一**：每个 Agent 只做一件事，并做到极致

* **质检前置**：研究报告经双重核查（锦衣卫 + 东厂）才进入大纲环节

* **人工最小化**：仅 2 个关键节点需要用户介入，其余全自动
